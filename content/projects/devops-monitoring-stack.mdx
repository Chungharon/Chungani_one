## Problem

Our team was flying blind on a suite of microservices. Errors surfaced only when
customers reported them, and diagnosing the root cause meant grepping through
logs manually. We needed structured metrics, persistent dashboards, and proactive
alerting before issues reached users.

## Approach

All components run as Docker containers orchestrated by Docker Compose, deployed
to a GCP Compute Engine instance behind a Cloud Load Balancer.

```
Services → Prometheus (scrape) → Alertmanager → Slack / Email
                ↓
            Grafana (visualize)
```

**Prometheus** scrapes metrics from every service on a 15-second interval with
a 30-day retention window. **Grafana** dashboards are provisioned via config
files — no manual setup after a redeploy. **Alertmanager** routes alerts to
`#ops-alerts` in Slack with severity labels and silencing rules.

Config changes trigger a GitHub Actions workflow that SSH-deploys to GCP and
reloads Prometheus live via `POST /-/reload` — zero downtime.

```yaml
groups:
  - name: service.rules
    rules:
      - alert: HighErrorRate
        expr: rate(http_requests_total{status=~"5.."}[5m]) > 0.05
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "High 5xx error rate on {{ $labels.service }}"
```

## Key Features

- RED method dashboards (Request rate, Error rate, Duration) per service
- Multi-severity alerts — warning at 90% disk, critical at 95%
- Automated Grafana provisioning — datasources and dashboards from config files
- TLS termination via Nginx + Let's Encrypt in front of Grafana
- Immutable image tags (`sha-abc1234`) for reproducible deployments

## Outcome

- **MTTD reduced from hours to minutes** — alerts fire within 2 minutes of threshold breach
- **On-call confidence improved** — dashboards give responders immediate context during incidents
- **Zero-downtime config updates** — live reload eliminates service restarts
- **Full observability** across 6 microservices from a single Grafana instance
